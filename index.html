Learning Log

4.16.2021

So because this is my first webpage, I figure this section might be a good place to explain some of the things that I'm learning while putting this together.

One of the first little quirks I ran into was while working with CSS to style the buttons. In this case, the buttons included the series at top of the page (Work, Personal, Etc), and the image buttons in the contact section on the bottom.

Part of what I was hoping to achieve was a certain amount of consistency in the hover effect, while needing to have different measures such as height and width maintained. So, my thought was to add the class of buttons to the profile images, which led to some interesting side effects.

As this this is the first time I've tried using multiple classes on a single HTML element, I was able to see the way that the cascade components are 'read' by the browser very quickly. As I had placed my images tag above my buttons tag, my image quickly shifted into square buttons due to the padding and spacing change. They were no longer affected by my border-radius which turned them into a circle and were instead sqaure images with a fair amount of padding.

I admit, I came to somewhat like the clearly broken style while I was working on other components. It certainly didn't look professional, but the consistency of the hover effects were pretty nice. It wasn't until later that the obvious solution to the problem occurred to me: put the images class below the button class and you can solve your problem as the cascade over-writes itself as it moves through the document. Immediately, I could have the styling for both sets of buttons and the same hover effect.

4.25.2021

While working on the JavaScript functionality of this page, one data structure that I have leaned heavily on for all three of my projects has been arrays. The ability to feed information into a function behind the scenes is truly invaluable and something that I had not grasped fully while simply studying and learning to code by playing around with basic funcionality. This has given me a small taste of what it means when something is scalable.

In the case of the calculator, I built a function which creates and appends all of the buttons by accessing the indices of an array and then assigns them object keys and values based on the expected functionality. While the makeCalc function took awhile to build (and it desperately needs to be broken up into smaller pieces for legibility and easier code maintenence), building it in this way allowed me to simply add two operators as strings (the exponent and modulo/remainder) to the array. With no additional code beyond that, including the function itself, they were immediately working and appended to the appropriate location. Similarly, after adding the decimal point to the array, it required only a single OR operator to be added to an if statement before it too was fully operational

While that was the most powerful example of array functionality and scaling I cam across, I also rediscovered it in smaller ways in the To-Do List and the Shakespeare Insult Generator. In the case of the To-Do list, an array enables the 'success statements' to fire when an item is clicked. For the insult generator, it's an easy numbers game: three arrays, two functions, and one event listener allow for 125,000 possible insults with very minimal code. In both cases, simply adding to the arrays and in no way touching any other text in the .js or .html files could expand the number of statements that could be generated and displayed to a user with a few keystrokes and git commands. Ultimately, I look forward to discovering the many ways that arrays are able to make functions scalable and adapatable to new features or increased capacities with ease.
